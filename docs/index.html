SaldeoSMART MCP Server

A Model Context Protocol (MCP) server that lets any MCP-aware client (Claude Desktop, Claude Code, Cursor, Zed, MCP Inspector, custom SDKs) read and write accounting data in SaldeoSMART: companies, contractors, documents, invoices, bank statements, dimensions, categories, personnel records — without you ever touching the underlying gzip-base64-XML API.

What you get out of the box. 26 typed tools (13 read + 13 write/merge), shipped as a Docker image and runnable via uvx. Drop one entry into your client's MCP config and the LLM behind it can answer "what cost invoices did Acme send last month?", file a new contractor, or push booking numbers from your ERP back to SaldeoSMART.

What it does

  • Wraps every flat REST endpoint of the SaldeoSMART API — versions 1.0 through 3.1 — as a Python function with typed inputs and Pydantic outputs.
  • Hides API quirks: MD5 request signing, XML→gzip→base64 encoding, the no-concurrent-requests rule, the per-item error envelope, gzip response decompression.
  • Speaks MCP over stdio, the protocol every MCP-aware client uses.
  • Stays read-only by default at the smoke-test layer, so a misconfigured CI run can never mutate production accounting data.

Who it's for

  • Bookkeepers using an LLM copilot — "find the missing supplier invoice from last week."
  • ERP integrators who need to push numbering back into SaldeoSMART after booking.
  • Anyone scripting SaldeoSMART access in Python — the underlying SaldeoClient class works without MCP.

Install & run

Build the Docker image

git clone <this-repo>
cd saldeosmart-mcp
make build
# = docker build -f docker/Dockerfile -t saldeosmart-mcp:latest .

The image is two-stage: uv resolves dependencies in a builder layer, then a slim python:3.12-slim-bookworm runtime inherits the prebuilt .venv. Final image size: ~234 MB.

Run it

export SALDEO_USERNAME=your-login
export SALDEO_API_TOKEN=your-token
make run
# = docker run --rm -i -e SALDEO_USERNAME -e SALDEO_API_TOKEN \
#       -e SALDEO_BASE_URL saldeosmart-mcp:latest

The server starts and waits for MCP messages on stdin/stdout. The -i and --rm flags are required: MCP uses stdio, so the container must have stdin attached, and there's no point keeping stopped containers around after the session.

Without Docker (local dev)

uv sync --extra dev
uv run saldeosmart-mcp     # console script declared in pyproject.toml

With the MCP Inspector

make inspector
# wraps:  npx @modelcontextprotocol/inspector docker run --rm -i ...

Wire it into your MCP client

The server speaks MCP over stdio, so any MCP-aware client can launch it. Pick a transport (Docker or uvx), then drop the appropriate snippet into your client's config.

{

      "command": "uvx",
      "args": [

Restart the app afterwards. If it complains it can't find docker / uvx, give it the absolute path from which docker / which uvx instead of the bare name.

Zed and other MCP clients

The launch command and args above work in any MCP-aware client. Consult the client's docs for where to put the JSON / where to invoke add.

Keep credentials out of the config file. Drop them into ~/saldeosmart.env (chmod 600) as KEY=VALUE lines, then replace the inline -e flags with "--env-file", "/Users/you/saldeosmart.env" in the Docker form. Easier to rotate, easier to keep out of source control.

Read tools

Read tools never mutate state and are safe to call from any context. All return Pydantic objects (which FastMCP serializes to JSON for Claude).

list_companies

read GET /api/xml/1.0/company/list

List companies (clients of the accounting firm). Every other tool needs a company_program_id — start here to find one.

list_contractors

read GET /api/xml/1.23/contractor/list

List contractors (suppliers and customers) for a company.

list_documents

read GET /api/xml/2.12/document/list

Fetch cost documents with one of three policies: LAST_10_DAYS (all recent), LAST_10_DAYS_OCRED (only OCR'd), or SALDEO (only those marked for export).

search_documents

read POST /api/xml/1.8/document/search

Find a single document by ID, document number, contractor NIP, or GUID. Combine fields to narrow it down.

list_invoices

read GET /api/xml/1.20/invoice/list

List sales invoices issued in SaldeoSMART (the invoicing module — distinct from cost documents).

list_bank_statements

read GET /api/xml/2.18/bank_statement/list

Bank statements with their operations, dimensions, and matched contractors. Requires the Bank Statement module on the account.

list_employees

read GET /api/xml/2.20/employee/list

Employees registered in the SaldeoSMART Personnel module.

list_personnel_documents

read POST /api/xml/2.20/personnel_document/list

HR documents (contracts, declarations). Filter by employee, year, month, or "only documents not yet exported."

get_document_id_list

read POST /api/xml/3.0/document/getidlist

(API 3.0) Discover document IDs for a folder, grouped by kind: contracts, cost invoices, sale invoices, etc. Pair with get_documents_by_id.

get_documents_by_id

read POST /api/xml/3.0/document/listbyid

(API 3.0) Fetch full document records for IDs you discovered earlier. Same response shape as list_documents.

get_invoice_id_list

read POST /api/xml/3.0/invoice/getidlist

(API 3.0) Invoice IDs grouped by kind: invoices, corrective invoices, pre-invoices, corrective pre-invoices.

get_invoices_by_id

read POST /api/xml/3.0/invoice/listbyid

(API 3.0) Fetch invoice records by ID.

list_recognized_documents

read POST /api/xml/2.18/document/list_recognized

Fetch OCR-processed data for a list of OCR origin IDs returned by recognize_documents. Saldeo requires at least one ID — there's no "list everything" mode.

Write tools

These tools mutate the customer's accounting data. Each call reaches the live SaldeoSMART account behind the configured credentials. Saldeo answers STATUS=OK at the envelope level even when individual items fail; every write tool returns a MergeResult with per-item statuses so callers can detect partial successes.

merge_contractors

write POST /api/xml/1.23/contractor/merge SS02

Add or update contractors in bulk. Supports nested bank accounts and email lists. Identify an existing contractor with contractor_program_id (your ERP ID) or contractor_id (Saldeo's ID).

merge_categories

write POST /api/xml/1.0/category/merge SS09

Sync the document categories used to classify cost documents.

merge_payment_methods

write POST /api/xml/1.0/payment_method/merge SS11

Sync payment methods (cash, transfer, card, …).

merge_registers

write POST /api/xml/1.0/register/merge SS10

Sync accounting registers.

merge_descriptions

write POST /api/xml/1.13/description/merge SS14

Sync the canned description list used on documents (e.g. "goods purchase", "fuel purchase").

merge_dimensions

write POST /api/xml/1.12/dimension/merge SS12

Sync analytical dimensions. type=ENUM accepts a fixed list of values; NUM/LONG_NUM/DATE accept free-form input.

merge_articles

write POST /api/xml/1.14/article/merge SS21

Sync the article catalog. Each article can carry per-supplier foreign codes.

merge_fees

write POST /api/xml/1.13/fee/merge SSK04

Push the accounting firm's fees for a specific month (folder = year + month).

merge_document_dimensions

write POST /api/xml/1.13/document_dimension/merge SS20

Set dimension values on existing documents. The dimensions referenced must already exist (call merge_dimensions first).

update_documents

write POST /api/xml/2.4/document/update SS17

Edit existing documents — number, dates, contractor, bank account. Only set the fields you want to change; None means "leave alone".

delete_documents

write POST /api/xml/1.13/document/delete SS16

Delete documents by ID. Destructive.

recognize_documents

write POST /api/xml/1.20/document/recognize SS06

Trigger OCR on previously-uploaded documents. Returns OCR origin IDs for use with list_recognized_documents.

sync_documents

write POST /api/xml/1.13/document/sync SS13

Push accounting numbering and status (BUFFER / INTRODUCED / BOOKED) back into Saldeo. Used by ERP integrations after the document is booked on the accounting side.

Input/output models

Every tool's signature uses Pydantic models, so FastMCP publishes a precise JSON Schema to Claude. A few examples:

Read response: DocumentList

class DocumentList(BaseModel):
    documents: list[Document]   # full document records
    count: int                  # convenience for "how many"

Write input: ContractorInput

class ContractorInput(BaseModel):
    short_name: str                              # required
    full_name: str                               # required
    contractor_program_id: str | None = None     # your ERP ID (for matching)
    contractor_id: int | None = None             # Saldeo's ID (alternative)
    supplier: bool | None = None
    customer: bool | None = None
    vat_number: str | None = None
    city: str | None = None
    postcode: str | None = None
    street: str | None = None
    country_iso3166a2: str | None = None
    telephone: str | None = None
    contact_person: str | None = None
    description: str | None = None
    payment_days: int | None = None
    bank_accounts: list[BankAccountInput] = []
    emails: list[str] = []

Write outcome: MergeResult

class MergeResult(BaseModel):
    operation: str | None       # echoed Saldeo operation name (METAINF/OPERATION)
    total: int                  # how many items the request contained
    successful: int              # items that came back with an OK per-item status
    errors: list[ItemErrorPayload] = []   # per-item failures (validation + operational)

The other input models (CategoryInput, RegisterInput, DimensionInput, ArticleInput, FeeInput, DocumentUpdateInput, DocumentSyncInput, RecognizeOptionInput, DocumentDimensionInput, …) follow the same shape: required fields are unannotated, optional ones default to None or empty list.

Error handling

Saldeo errors come in three flavors. The MCP layer normalizes them into a single ErrorResponse shape Claude can reason about.

ClassWhat it looks likeHow we surface it
Top-level error <RESPONSE><STATUS>ERROR</STATUS><ERROR_CODE/><ERROR_MESSAGE/></RESPONSE> (HTTP 200, the body holds the truth) ErrorResponse(error=ERROR_CODE, message=ERROR_MESSAGE)
HTTP transport error Non-2xx response with no structured body, or a network/parse error. ErrorResponse(error="HTTP_500" | "PARSE_ERROR" | "NETWORK_ERROR", …)
Per-item failure STATUS=OK at the envelope, but individual items have UPDATE_STATUS=NOT_VALID or STATUS=ERROR with nested <ERRORS>. MergeResult.errors (list of ItemErrorPayload).

Common error codes you'll actually see:

  • 4001 — invalid signature (usually a credentials mistake)
  • 4302 — user is locked
  • 4401 — bad request body (e.g. wrong tag name)
  • 6001 — feature not enabled on the account (Bank Statements, Personnel)

The package is organized as a strict stack. A higher layer may import from layers below it, never the other way around. A test (tests/unit/test_architecture.py) parses every import statement in the source tree and fails CI on any upward dependency.

src/saldeosmart_mcp/
├── config.py          # SaldeoConfig — env-driven settings
├── errors.py          # SaldeoError, ErrorResponse, ItemError, MergeResult,
│                      # iter_item_errors (per-item failure walker)
├── logging.py         # daily-rotated file logger
│
├── http/              # transport layer
│   ├── signing.py     # RequestSigner — MD5(URL-encode(sorted) + token)
│   ├── client.py      # SaldeoClient — httpx + threading.Lock + envelope parser
│   └── xml.py         # el_text/el_int/el_bool/set_text + URL redaction
│
├── models/            # everything serialized over MCP
│   ├── common.py      # cross-resource (BankAccount, BankAccountInput)
│   ├── companies.py   ├── contractors.py
│   ├── documents.py   ├── invoices.py
│   ├── bank.py        ├── personnel.py
│   └── catalog.py     # categories, registers, payment methods, articles, fees
│
├── tools/             # @mcp.tool registry — one file per resource
│   ├── _runtime.py    # mcp = FastMCP(), saldeo_call, get_client,
│   │                  # summarize_merge, parse_collection
│   ├── _builders.py   # XML builders shared across resources
│   ├── companies.py   ├── contractors.py   ├── documents.py
│   ├── invoices.py    ├── bank.py          ├── personnel.py
│   ├── dimensions.py  └── catalog.py
│
└── server.py          # main() — entrypoint; imports tools, runs mcp.run()

Why this layout

  • One file per Saldeo resource family. Looking for the document tools? tools/documents.py. Bank statements? tools/bank.py. The package mirrors how the API itself is organized.
  • Strict layering. config → errors → http → models → tools → server. Lower layers know nothing about upper ones, so swapping any one layer is contained.
  • The mcp instance lives in tools/_runtime.py. Every tool module imports it once and registers @mcp.tool decorators against it. tools/__init__.py imports every submodule so the registry is populated as soon as saldeosmart_mcp.tools is imported.
  • server.py is a 40-line entrypoint. Logging setup, tool import (registers everything), then mcp.run(). Nothing else.

Request signing

SaldeoSMART rejects every request without three auth params: username, req_id, req_sig. The signature algorithm is the most error-prone part of the API contract — it's encapsulated in the RequestSigner class:

1. Take all params (URL + form, except `req_sig` itself).
2. Sort alphabetically by key.
3. Concatenate as `key=value` with NO separator between pairs.
4. URL-encode the result (space → '+', not %20).
5. Append the API token.
6. MD5 the bytes, hex digest.

We pass usedforsecurity=False to hashlib.md5() so FIPS-mode systems and Bandit don't flag the call — the spec mandates MD5; it's not a security choice.

The token itself is held as a Pydantic SecretStr, so accidental repr() in logs or error frames shows SecretStr('**********'), never the raw value.

Concurrency & rate limits

SaldeoSMART's spec says:

  • 20 requests per minute per user
  • No concurrent requests — wait for the previous response before sending the next
  • Max payload: ~70 MB after base64 encoding

FastMCP can dispatch tools concurrently from a thread executor, which would violate the second rule. The client wraps every HTTP call in a threading.Lock so calls serialize at the client boundary, even if Claude triggers multiple tools at once.

Rate limiting is not enforced client-side today; if you expect bursty traffic, add a token-bucket or use a server-side cache.

Logging

Logs land in a daily-rotated file under SALDEO_LOG_DIR (default ~/.saldeosmart/logs/saldeosmart.log; in the Docker image, /var/log/saldeosmart/saldeosmart.log). Retention is SALDEO_LOG_RETENTION_DAYS days (default 7).

Why a file, not stdout? MCP uses stdio as the transport. Anything written to stdout would corrupt the protocol stream; only a file handler is attached.

Every API call logs at INFO with the operation name parsed from the response's METAINF. Errors log at WARNING with code, message, HTTP status, and a redacted URL — req_sig and api_token are scrubbed from the URL before logging.

# Example log line for a successful list_documents call:
2026-04-28 13:21:55 INFO saldeosmart_mcp.client: SaldeoSMART OK: \
    operation=document.list \
    url=https://saldeo.brainshare.pl/api/xml/2.12/document/list?company_program_id=...&req_sig=***

To keep logs outside the container, mount a volume:

docker run --rm -i \
    -e SALDEO_USERNAME=... -e SALDEO_API_TOKEN=... \
    -v saldeosmart-logs:/var/log/saldeosmart \
    saldeosmart-mcp:latest

Testing & smoke test

Two layers of testing:

Unit tests (no network)

make test
# 130 tests covering: signing, gzip+base64 round trip, error envelope parsing
# (top-level + per-item), URL redaction, request lock concurrency,
# SecretStr token protection, every request-XML builder, package layering.

Tests mirror the source layout — the home for a new test is wherever the production code it covers lives:

tests/
├── conftest.py                       # shared fixtures (isolated_root_logger, clean_env)
└── unit/

    ├── test_errors.py                # SaldeoError → MCP payload shape
    ├── test_logging.py               # setup_logging env-var handling, rotation
    ├── http/
    │   └── test_client.py            # signing, gzip+base64, error-envelope parser
    ├── models/
    │   └── test_documents.py         # XML → Pydantic for documents/invoices
    └── tools/
        └── test_builders.py          # every request-XML builder + summarize_merge

Live smoke test

Drop a .env at the repo root:

SALDEO_USERNAME=your-login
SALDEO_API_TOKEN=your-token
SALDEO_BASE_URL=https://saldeo.brainshare.pl

Then:

uv run python scripts/smoke_test.py

It walks every read tool against the real account, prints a per-call pass/fail, and exits non-zero on failures.

The smoke test is read-only by policy. It never invokes merge_*, update_*, delete_*, recognize_*, sync_*, or any other write tool — credentials in .env point at a real production account, and live tests must not mutate it. Write tools are covered exclusively by unit tests against fixture XML.

Troubleshooting

Claude says "I don't have access to that tool"

  • Restart Claude Desktop after editing claude_desktop_config.json.
  • If docker isn't on Claude's PATH, supply the absolute path from which docker instead of the bare name.
  • Check Claude's MCP log (Settings → Developer → MCP logs) for startup errors.

Error 4001 "Invalid signature"

Almost always a credentials problem. Verify SALDEO_USERNAME and SALDEO_API_TOKEN in the container — try docker run --rm -it saldeosmart-mcp:latest /bin/sh and echo $SALDEO_USERNAME.

Error 6001 "User does not have access"

The endpoint requires a SaldeoSMART module that isn't enabled on the account (typical: Bank Statements, Personnel). Contact api@saldeosmart.pl if you think you should have it.

Error 4401 "No SEARCH_POLICY found in file"

Used to be a bug in older builds — the request XML emitted <POLICY> instead of <SEARCH_POLICY>. Fixed in commit cf92698; if you see it now, you're running an outdated image.

What's not yet covered

These endpoints exist in the SaldeoSMART REST API but aren't wrapped as MCP tools yet — the common factor is that they require uploading file attachments (gzip+base64 blobs in attmnt_X form fields):

  • document.add (SS05)
  • document.add_recognize (AE01) and document.correct (AE02)
  • document.import (3.0)
  • declaration.merge (SSK02)
  • assurance.renew (SSK03)
  • financial_balance.merge (SSK01) — supports optional attachments
  • invoice.add (3.0/3.1, SSK06)
  • employee.add (P03)
  • personnel_document.add (P04)
  • company.create (SS01) and company.synchronize (SS15) — many required fields, low value for an interactive Claude session

The lower-level SaldeoClient.post_command(..., extra_form={"attmnt_1": base64_blob}) already accepts attachments, so adding these as tools is a follow-up rather than a redesign.
